## Findings

### FINDING-1: A2A Messages Lack Sender Authentication [MEDIUM]

**Package:** `internal/a2a/a2a.go`
**Risk:** Any agent can set `msg.From` to any arbitrary ID. The `ServeHTTP` handler accepts inbound messages without verifying the `From` field matches the actual sender. An agent could impersonate another agent.

**Evidence:**
```go
// HandleMessage processes an incoming message — no From verification
func (s *A2AServer) HandleMessage(msg Message) (Message, error) {
    handler, ok := s.handlers[msg.Type]  // trust msg.From blindly
```

**Impact:** Agent impersonation, privilege escalation via forged admin messages, false audit trails.

**Recommendation:** Add message signing (HMAC or Ed25519). Each agent gets a key pair; messages include a `Signature` field. `HandleMessage` verifies signature matches `From` before processing.

### FINDING-2: Relay Messages Lack Integrity Verification [LOW-MEDIUM]

**Package:** `internal/relay/relay.go`
**Risk:** The relay's `Send()` accepts any `From` field without verification. In a pub/sub model, a compromised agent could publish messages claiming to be another agent. Dead letters carry the same risk.

**Evidence:**
```go
func (r *Relay) Send(msg Message) (*Message, error) {
    // msg.From is taken as-is, no verification
    r.allMessages[msg.ID] = &msg
```

**Impact:** False commands, social engineering between agents, audit trail pollution.

**Recommendation:** Require message authentication codes (MACs) on relay messages. Agents register with a shared secret or public key; relay verifies before enqueuing.

### FINDING-3: Tenant Role Comes From Client Header [MEDIUM]

**Package:** `internal/tenant/middleware.go`
**Risk:** `RequireRole` reads role from `X-User-Role` HTTP header. An attacker can set this header to "owner" or "admin" and bypass role checks entirely.

**Evidence:**
```go
func (m *Middleware) RequireRole(minRole Role, next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        roleStr := r.Header.Get("X-User-Role")  // client-controlled!
        if roleStr == "" {
            roleStr = "viewer"
        }
        role := Role(roleStr)
        if !roleAtLeast(role, minRole) {
```

**Impact:** Full privilege escalation — any user can become owner/admin by setting a header.

**Recommendation:** Derive role from authenticated session/token, not from headers. The role should come from a verified JWT claim or a server-side lookup based on the authenticated user ID.

### FINDING-4: Sandbox Has No True Isolation Mechanism [LOW]

**Package:** `internal/sandbox/sandbox.go`, `internal/boundary/boundary.go`
**Risk:** Sandboxed code runs as the same OS user with the same permissions. `IsolationNetwork`/`IsolationFull` attempt to use `httpjail` but fall back to no-op if unavailable. Filesystem isolation sets env vars but doesn't enforce actual access control. UID/GID isolation is a no-op placeholder.

**Evidence:**
```go
// boundary.go — user isolation is a no-op
func applyUserIsolation(cmd *exec.Cmd, config Config) {
    _ = config.UID
    _ = config.GID  // intentionally unused
}
```

**Impact:** Sandboxed agents can read/write any file the host user can, access network freely, and execute arbitrary commands.

**Recommendation:** For production: use gVisor, Firecracker microVMs, or Docker containers with dropped capabilities. At minimum: use Linux namespaces (user, mount, network) via `unshare` or `bwrap` (bubblewrap). The current implementation is suitable for development only.

### FINDING-5: Ledger Budget Bypass Possible on Boundary [LOW]

**Package:** `internal/ledger/ledger.go`
**Risk:** Budget enforcement happens at `RecordUsage` time. If an agent bypasses the ledger (calls the model API directly without going through `RecordUsage`), the budget is not enforced. The ledger is a library, not a proxy — it trusts callers to use it.

**Evidence:**
```go
func (l *Ledger) RecordUsage(...) error {
    if l.budget > 0 && l.totalCost+costUSD > l.budget {
        return fmt.Errorf("budget exceeded...")
    }
```

**Impact:** Agents can exceed budget by calling model APIs without recording usage.

**Recommendation:** Add a model API proxy layer that enforces budget before forwarding requests. All model calls must go through this proxy — the ledger becomes the gate, not just the accountant.
