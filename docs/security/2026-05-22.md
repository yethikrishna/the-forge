## Findings by Domain

### F1. Organization Isolation вАФ MEDIUM

**Tenant data is scoped by tenant ID, but enforcement is not at the data layer.**

- `tenant.Manager` stores all tenants in a single `map[string]*Tenant`. Access is by tenant ID lookup. вЬЕ Correct.
- `tenant.Middleware.RequireTenant` extracts `X-Tenant-ID` from header/query. вЬЕ Present.
- **GAP**: There is no per-tenant data isolation in the `org`, `ledger`, `consent`, `coordination`, or `knowledge` packages. These packages store data in flat maps keyed by agent/division IDs, but there is no tenant/org ID filter. If multiple tenants share a single Forge instance, one tenant's agent data is visible to another.
- **GAP**: The `org.Org` struct is a single-org structure. There is no `orgID` or `tenantID` field. If multiple orgs exist, they have separate `Org` instances вАФ which is correct isolation by separate instances, but there is no enforcement that an agent in org A cannot reference data from org B.

**Recommendation**: Add `tenantID` to all data-bearing structs and enforce it in queries. At minimum, document that each tenant must run a separate process with separate data directories.

### F2. Agent Identity & Impersonation вАФ HIGH

**A2A protocol has NO authentication. Agents can impersonate each other.**

- `a2a.Message` has a `From` field set by the sender. There is no signature, MAC, or token verification on the receiving end.
- Any agent can set `From: "agent-CEO"` and send commands as another agent.
- The `AgentCard` has an `AuthSchemes` field but it is informational вАФ never validated.
- `SendMessage` uses plain HTTP POST with no TLS requirement and no auth headers.

**Recommendation**: Add HMAC-SHA256 message signing. Each agent gets a signing key at registration. Receivers verify `From` against the signature. Add mTLS or at minimum Bearer token auth for all A2A HTTP endpoints.

### F3. Cost Controls вАФ MEDIUM

**Budget checking is advisory, not enforced.**

- `cost.Tracker.Record()` records usage and estimates cost but does NOT check budget or reject over-budget requests. The method signature `Record(...) (*UsageRecord, error)` returns `(record, nil)` even when over budget.
- `CheckBudget()` exists and returns `BudgetStatus.OverBudget = true` when exceeded, but callers are not required to call it before `Record()`.
- `RecordUnchecked()` explicitly bypasses even the advisory check.
- `tenant.CheckQuota()` in the tenant middleware checks quota, but only at the HTTP layer. Internal Go API calls skip it.
- `ledger.Ledger.SetBudget()` sets a budget and `Record()` fires alerts, but does not prevent the record from being appended.

**Recommendation**: Make `cost.Tracker.Record()` return an error when budget is exceeded. Add a `RecordEnforced()` that blocks over-budget usage. Gate all LLM API calls through budget check.

### F4. Compliance Gates вАФ LOW

**Compliance controls are present and well-structured.**

- `consent` package has hash-chain integrity (SHA-256 checksums with prev-hash linking). вЬЕ
- `ledger` package has hash-chain integrity for all entries. вЬЕ
- `audit` and `auditlog` packages log all agent actions. вЬЕ
- `compliance` package maps controls to SOC2/HIPAA/GDPR/ISO27001. вЬЕ
- **GAP**: Compliance report generation is manual (`Generate()`). No automatic gate that blocks regulated actions without consent.

**Recommendation**: Add a middleware/interceptor that checks consent status before allowing data processing for regulated purposes.

### F5. Feedback Loops вАФ LOW

**Production signal paths exist but are not wired end-to-end.**

- `coordination.Tracker` detects stuck agents and auto-escalates. вЬЕ
- `org.Org.Escalate()` routes to division head. вЬЕ
- `feedback` package collects feedback. вЬЕ
- **GAP**: No connection from `coordination` stuck-detection вЖТ `org` escalation вЖТ `feedback` collection вЖТ division-level dashboard. Each component works independently.

**Recommendation**: Wire stuck events through the escalation pipeline and surface in the dashboard.

### F6. Memory Isolation вАФ MEDIUM

**Knowledge and memory packages lack org-scoped queries.**

- `knowledge` package stores entries in flat maps keyed by topic/ID.
- `memory` in OpenClaw bridge is file-based (`memory/*.md`).
- No `orgID` or `tenantID` filter on knowledge queries.
- An agent from org A querying knowledge could theoretically see org B's knowledge if they share the same process.

**Recommendation**: Prefix all knowledge keys with org/tenant ID. Enforce scope in query methods.

### F7. OpenClaw Bridge вАФ LOW

**Bridge is a well-structured HTTP client with proper auth.**

- `openclaw.Bridge` uses `Authorization: Bearer <token>` for gateway auth. вЬЕ
- Config path is auto-detected. вЬЕ
- Token is stored in bridge config, not hardcoded. вЬЕ
- **GAP**: No TLS verification config вАФ relies on default Go TLS. In production, consider pinning CA certs.
- **GAP**: `LoadConfig()` reads the full OpenClaw config including potentially sensitive gateway auth keys. Consider scoping to only needed fields.

### F8. Suna Bridge & Sandbox вАФ MEDIUM

**Sandbox isolation is Docker-based with resource limits. вЬЕ But has gaps.**

- Resource limits (CPU, memory, disk) are set via Docker flags. вЬЕ
- Network is set to `bridge` mode. вЬЕ
- `applyFileSystemIsolation` in boundary sets env vars but does not enforce at kernel level (needs root for chroot). вЪ†пЄП
- **GAP**: `SandboxManager.Exec()` runs `docker exec ... sh -c <command>` вАФ the command is shell-interpreted. If `command` contains user input, this is a shell injection vector.
- **GAP**: No seccomp/AppArmor profile specified for containers.
- **GAP**: Container names are predictable (`forge-sb-{agentID}-{name}`), making them discoverable.

**Recommendation**: Use `exec.Cmd` with argument array instead of `sh -c`. Add `--security-opt seccomp=...` and `--security-opt apparmor=...` to Docker run. Use random suffixes in container names.

### F9. Quality Gates вАФ MEDIUM

**Quality gates exist but are not enforced in the agent pipeline.**

- `govern` package scores agents on quality, cost, speed, reliability. вЬЕ
- `approval` package provides human-in-the-loop gates. вЬЕ
- `guard` package provides safety guardrails. вЬЕ
- **GAP**: Agent execution does not require passing quality gates before producing output. An agent can produce low-quality output and it flows to downstream agents without quality filtering.
- **GAP**: Quality scores are computed after execution, not checked before.

**Recommendation**: Add a quality gate check in the handoff pipeline вАФ agents must meet minimum quality thresholds before their output is accepted by downstream agents.

### F10. Trust Verifiability вАФ LOW

**Claims are independently verifiable via hash chains. вЬЕ**

- Ledger entries are hash-chained: each entry's hash includes the previous hash. вЬЕ
- Consent records are hash-chained. вЬЕ
- `Verify()` methods exist on both `Ledger` and `ConsentManager`. вЬЕ
- Audit logs are append-only with timestamps. вЬЕ

**GAP**: No external attestation mechanism вАФ a compromised Forge process could rewrite both the data and the hashes. Consider external witness/notary.

## Critical Bug Found (Not Security)

### StuckStatus zero-value bug (`internal/coordination`)

`var status StuckStatus` defaults to `""` (empty string), which does NOT equal `StuckNone = "none"`. This caused ALL agents to be detected as stuck immediately upon their first progress report, with empty status and reason. This would have caused false escalation spam in production.

**Severity: HIGH** (functional, not security)
**Impact**: All coordination stuck-detection was broken since introduction.
**Fix**: Initialize `status` to `StuckNone`.
